# Contributing to ranbval-sdk

Thank you for considering a contribution. This document explains how to get set up, how the project is structured, and what we look for in pull requests.

---

## Prerequisites

- **Node.js 18 or later** ŌĆö the SDK targets Node 18 and uses `node:test`, `node:crypto`, and other built-in modules that require it.
- **npm** ŌĆö used for dependency management and scripts. No other package manager is required.

---

## Getting started

```bash
git clone https://github.com/TariqDreamsTech/ranbval-sdk-node.git
cd ranbval-sdk-node
npm install
npm test
```

`npm install` only pulls in `esbuild` (a dev dependency). There are no production dependencies.

---

## Project layout

```
src/          Source files ŌĆö the only code that matters for the published package
  index.js    Main entry point; re-exports everything public
test/         Test files using node:test
  *.test.js
examples/     Runnable examples (not published to npm)
  basic.js
dist/         Built output ŌĆö generated by `npm run build`, do not edit by hand
```

All public exports live in `src/index.js`. Internal modules are in `src/` as well. The `dist/` folder is produced by esbuild and checked in only for release commits.

---

## Building

```bash
npm run build
```

This runs `esbuild` to bundle `src/index.js` into a single CommonJS file at `dist/index.js`. The build is minified and has a one-line license comment prepended automatically.

You do not need to rebuild to run tests ŌĆö `npm test` runs the source files directly via `node --test`.

---

## Running tests

```bash
npm test
# or
node --test test/*.test.js
```

Tests use Node's built-in `node:test` runner. No Jest, Mocha, or other test framework is needed.

### Adding tests

- Add a new file to `test/` named `*.test.js`.
- Use `node:test` and `node:assert` from the standard library.
- Keep each test file focused on one module or feature area.
- Tests must not make real network calls. Mock external HTTP calls using a local `http.createServer` or by patching the relevant function with a stub.
- Tests must not write to `process.env` without restoring original values in a cleanup hook.

Example test skeleton:

```js
const { test } = require('node:test');
const assert = require('node:assert/strict');
const { safeDecrypt } = require('../src/index.js');

test('safeDecrypt rejects a malformed token', async () => {
  await assert.rejects(
    () => safeDecrypt('not-a-token', 'any-secret'),
    TypeError
  );
});
```

## Code style

- **No runtime dependencies.** If you find yourself reaching for a third-party package, find a way to use Node built-ins instead. This is non-negotiable ŌĆö the zero-dependency guarantee is a core feature.
- **CommonJS only.** The published bundle is CJS. Source files use `require`/`module.exports`. Do not use top-level `import`/`export` syntax.
- **ES2020 syntax is fine** in source files ŌĆö esbuild transpiles for Node 18 as a target.
- **No transpiled TypeScript in `src/`.** JSDoc comments are welcome for editor tooling, but `.ts` source files are not part of this project's current build pipeline.
- **Keep functions small and focused.** Side-effect-free helpers are easier to test.
- **Wipe secrets.** Any code path that touches plaintext key material must call `.wipe()` or ensure the `SecretString` is not retained beyond its use.

## Pull request guidelines

1. **Open an issue first** for anything larger than a bug fix or a small documentation change. This avoids wasted effort if the direction doesn't align.
2. **One concern per PR.** A PR that mixes a bug fix with a refactor and a new feature is hard to review. Split them.
3. **All tests must pass.** Run `npm test` before pushing. CI will also run tests, but it's faster to catch failures locally.
4. **Add tests for new behavior.** PRs that add or change behavior without accompanying tests will be asked to add them before merging.
5. **Update the changelog.** Add an entry under `[Unreleased]` in `CHANGELOG.md` describing your change.
6. **Do not bump the version.** The maintainers handle version bumps and releases.

---

## Reporting issues

Use the [GitHub issue tracker](https://github.com/TariqDreamsTech/ranbval-sdk-node/issues). Please include:

- Node.js version (`node --version`)
- A minimal reproduction ŌĆö a short code snippet and the error output is ideal.
- What you expected to happen vs. what actually happened.

For security vulnerabilities, do not open a public issue. Contact the maintainers directly.
